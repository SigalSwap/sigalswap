# SigalSwap and Aztec-runtime subtleties

Three Aztec-runtime edge cases that affect SigalSwap end-user UX and SDK behavior. Each is researched against v4.3.0 source and Aztec's published architecture; each is tagged **verified**, **probably true (unverified)**, or **roadmap-dependent** in the spirit of `docs/mev-considerations.md`.

The three concerns:
1. **PXE oracle data freshness** — what happens when the user's PXE is one or more blocks behind?
2. **Block reorgs and finality** — can a swap appear committed in the SDK then revert?
3. **Authwit replay across failures** — if a tx fails partway through, is the authwit consumed?

## 1. PXE oracle data freshness

### The model (verified against v4.3.0 source + Aztec docs)

Private functions on Aztec execute in the user's **PXE** (Private eXecution Environment). The PXE maintains a synchronized view of the L2 chain via a `BlockSynchronizer` that polls the Aztec node for new blocks. Private execution **anchors** at a specific block number: the proof references the state root at that anchor block, and any public state the private function reads is read **at the anchor block's state**, not at execution time.

When a private function reads public state (e.g., the pair's `packed_reserves`), the value used is the value at the anchor block. The Aztec protocol then validates at sequencer-side inclusion that the anchor block is recent enough to be acceptable (within a protocol-defined staleness window).

Relevant aztec-nr source (`noir-projects/aztec-nr/aztec/src/context/private_context.nr`):
- `get_block_header_at(block_number)` retrieves a block header — the comment notes "`block_number` ... must be `<=` anchor block number."
- The anchor model is structural to how Aztec PXE delivers zero-knowledge proofs at all.

### What this means for SigalSwap swaps

A private swap call (e.g., `pair.swap_exact_in(...)`) does the following inside the user's PXE:
1. Reads the immutable `config` and validates the token pair.
2. Transfers the input token and prepares one partial note for the output.
3. Emits the wallet-history event and enqueues the public continuation.

The private leg does **not** read `packed_reserves` and does **not** call `get_amount_out`. All swap math — the reserve read, `get_amount_out`, and the `amount_out_min` check — happens in the **public continuation** (`_swap_exact_in`), which runs on the sequencer at inclusion time. It reads the current `packed_reserves` and computes the amounts against live state. If reserves have moved since the user signed, the public computation yields a different `amount_out` — and the user's `amount_out_min` floor reverts the tx cleanly if the new amount is below the floor.

So **stale PXE → graceful slippage revert** if other activity has moved the price during the staleness window. Because the amount math runs against live reserves in the public phase rather than against the anchor-block snapshot, the user's slippage tolerance is the structural protection. The contract enforces this at `protocol/periphery/src/main.nr:594` (router multi-hop) and equivalent sites in the pair.

### Recommendations

For wallets / SDK consumers:

- **Surface the anchor-block age** in the wallet UI (e.g., "your PXE is N blocks behind"). A staleness of more than 2-3 blocks should prompt a re-sync before signing.
- **Use wider slippage tolerance** for users on hosted PXE infrastructure that may lag.
- **The SDK's slippage helpers** (`packages/sdk/src/slippage.ts`) and the default tolerances in `packages/sdk/README.md` are the right primitive; nothing additional needed on the SigalSwap contract side.

### Verified vs. open

**Verified:** PXE anchors private execution at a specific block; public continuations re-read state at inclusion; SigalSwap's `amount_out_min` / `amount_in_max` are the graceful-failure mechanism.

**Open (for Aztec consultation):**
- What is the protocol-defined staleness window (max blocks between anchor and inclusion)?
- Are there PXE-side telemetry primitives wallets should expose to users when the anchor is approaching the staleness limit?

## 2. Block reorgs and finality

### Aztec finality model (verified against Aztec docs + Ignition launch material)

Aztec uses a **two-stage finality model**:

- **Pending Chain.** Blocks are proposed by the slot's Fernet-elected sequencer and attested to by the rotating committee (two-thirds-plus-one quorum). These blocks become part of a "pending" chain that is canonical to active sequencers but not yet final.
- **Proven Chain.** Pending-chain blocks "graduate" to the Proven Chain once the corresponding ZK validity proofs are generated by provers and verified on Ethereum L1. At that point, the blocks inherit Ethereum's settlement guarantees.

A tx appears as "mined" in the SDK once it's in the Pending Chain. **It is not yet final at that point.** Pending-chain reorgs are possible if:
- Proof generation fails or stalls.
- The L1 settlement transaction doesn't land in a reasonable window.
- The sequencer set encounters a fault that requires rollback.

In all such cases, Pending-chain blocks (and their txs) can be reverted before L1 finality.

Once a tx is in the Proven Chain (L1-settled), it inherits Ethereum reorg semantics — the same finality as any Ethereum L1 tx, which is "very strong" but not absolute (L1 reorgs of depth 1-2 are rare but possible; deep reorgs essentially impossible after a few minutes).

### What this means for SigalSwap

The SDK currently returns `Promise<TxSendResultMined>` from every tx-sending method (`packages/sdk/src/pair.ts`, `router.ts`). The aztec.js `TxSendResultMined` type resolves when the tx is **in a block** — which is Pending Chain inclusion. **It does NOT wait for L1 finality.**

For most user flows, this is fine: a Pending Chain inclusion gives the user immediate feedback and is the right UX. But there are scenarios where the difference matters:

- **High-value swaps** where the trader wants L1 finality before considering the trade "real."
- **Integrator contracts** that hand off post-swap actions (e.g., a vault that records SigalSwap output and then mints shares). These should distinguish "swap pending" from "swap final."
- **Indexers and analytics** that need to handle pending-block reverts cleanly.
- **External recipients of swap output** (e.g., centralized exchange deposits). CEX policy usually requires N L1 confirmations.

### Recommendations

For the SDK (future work, currently not implemented):

- **Add a finality-status query.** Expose `client.getTxStatus(txHash)` returning `'pending' | 'proven' | 'l1-finalized' | 'reverted'`. Wraps the underlying aztec.js node API.
- **Add an optional `waitForProven` helper.** Lets integrators block on L1 settlement when their use case requires it.
- **Document the Pending vs Proven distinction in the SDK README.** Integrators reading `TxSendResultMined` shouldn't assume L1 finality.

For SigalSwap contracts: no contract-level changes needed. Reorg handling is a property of the L2, not the protocol.

### Verified vs. open

**Verified:** Aztec has a Pending / Proven two-stage chain; SDK currently returns on Pending inclusion; reorgs are possible pre-Proven.

**Open (for Aztec consultation):**
- What's the typical / worst-case time from Pending inclusion to L1 finality on Ignition mainnet? Spread for proof generation + L1 inclusion.
- Are there protocol-level guarantees about how deep a Pending reorg can be (in blocks), or is it bounded only by "before L1 proof is verified"?
- What aztec.js APIs are recommended for distinguishing tx states beyond `mined`?

## 3. Authwit replay across failures

### The mechanic (verified against v4.3.0 source)

SigalSwap's private entries take an `authwit_nonce: Field` argument and forward it to the token / LP-token transfer functions they invoke. Those callees -- not the pair's own entries -- carry the `#[authorize_once("from", "authwit_nonce")]` macro, which calls `assert_current_call_valid_authwit(...)` to verify the user authorized the pair (or router) to move their tokens. The authwit is therefore enforced at the contract that actually debits the user (the token / LP-token); the pair's private entries thread the nonce through but do not themselves carry the macro.

Inside `noir-projects/aztec-nr/aztec/src/authwit/auth.nr:271-286`:

```
pub fn assert_inner_hash_valid_authwit(context: &mut PrivateContext, on_behalf_of: AztecAddress, inner_hash: Field) {
    // ...static call to verify the authwit...
    let nullifier = compute_authwit_nullifier(on_behalf_of, inner_hash);
    context.push_nullifier(nullifier);
}
```

The authwit nullifier is **pushed onto the tx's nullifier set at validation time**, inside the private function execution. The nullifier is computed from `(on_behalf_of, inner_hash)` where `inner_hash` covers the (caller, target-selector, args, nonce) tuple. Single-use is enforced by the nullifier set: once a nullifier is committed to L2 state, no future tx can re-emit it.

**The critical property: tx atomicity.** Aztec txs commit atomically. If any phase of the tx (private execution, public continuation, kernel proof, sequencer execution) reverts or fails, **no nullifier is committed**. The push at private-execution time is provisional; commitment to canonical state requires the whole tx to succeed.

Public-side authwits go through the `CANONICAL_AUTH_REGISTRY_ADDRESS.consume(...)` call (line 320-325 of the same file). Same atomicity guarantee: the registry's state update only commits if the tx commits.

### What this means for SigalSwap

- **Failed tx → authwit not consumed → user can retry** with the same authwit nonce. Verified.
- **Successful tx → authwit consumed → can't be replayed.** Verified.
- **No partial-state risk:** Aztec doesn't expose a state in which the authwit is consumed but the contract's intended side effects (token transfer, state update) didn't happen. Either both commit or neither does.

This means cross-tx authwit replay (the worry: "what if the tx fails after the authwit is consumed, leaving the user with a burnt nonce and no swap") **structurally cannot happen on Aztec**. The user always either gets the swap or gets to retry with the same nonce.

### Existing test coverage

`protocol/core-tests/src/test/security.nr` already includes three drift-probe canaries that pin the **fail-fast at entry** property — early-revert paths (zero input/output amounts) revert *before* any token-side call, so authwits aren't burned on stale-UI / automation-bug inputs:

- `test_swap_exact_in_private_zero_amount_in_reverts`
- `test_swap_exact_out_private_zero_amount_out_reverts`
- `test_swap_exact_out_private_zero_amount_in_max_reverts`

These verify the **entry-guard ordering** (boundary asserts run before authwit consumption on the offending paths). The deeper atomicity property — that an authwit pushed during private validation is *not committed* if the public continuation later reverts — is a framework-level Aztec guarantee rather than a SigalSwap defense. Every Aztec contract relies on it; if it failed, the entire L2 model would be broken. A SigalSwap-specific test for it would be testing Aztec's tx atomicity, not our code. The existing fail-fast canaries are the appropriate scope for SigalSwap-level coverage.

### Verified vs. open

**Verified:** Authwit consumption is atomic with the tx; failed txs do not consume authwits; the existing canaries verify entry-side fail-fast. The deeper public-continuation-revert atomicity is a framework-level Aztec guarantee (see above), not a separate SigalSwap canary — testing it would test Aztec's tx atomicity, not our code.

**Open (for Aztec consultation):**
- Is there any tx phase between "private validation complete" and "block inclusion" where partial state could persist (e.g., kernel intermediate states)? Standard answer is "no, kernels are atomic with the tx," but worth confirming for v4.3.

## Open questions for the Aztec team consultation

These sub-questions are largely answered by source review and public docs; the remaining items are open for the Aztec team consultation:

1. **Staleness window**: max number of blocks between PXE anchor and tx inclusion before sequencer-side rejection?
2. **Pending → Proven latency** on Ignition mainnet: typical, p95, worst-case in current operating regime.
3. **SDK API for finality status**: what's the recommended aztec.js path for tx state beyond `mined`?
4. **Reorg depth bound**: protocol guarantees on max Pending-chain reorg depth, or is it bounded only by L1 proof verification?
5. **Kernel atomicity confirmation**: no intermediate kernel states that could partially commit nullifiers?

SDK enhancements (finality-status query, `waitForProven` helper) are tracked for a future SDK release.
